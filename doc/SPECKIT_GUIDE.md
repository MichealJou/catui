# Speckit 项目规范工具 - 使用指南

**最后更新**: 2025-02-13

---

## 📋 什么是 Speckit？

**Speckit** 是一个项目规范管理工具，帮助团队：

- 📁 制定清晰的项目规范
- 🎯 追踪需求和决策
- 📝 生成规范化的文档
- 🔄 管理开发工作流程

---

## 🚀 快速开始

### 安装和配置

Speckit 通常需要配合项目配置文件一起使用。CatUI 项目已创建配置文件：

```bash
# 项目根目录
.specify/config.yaml
```

### 基本概念

#### 规范 (Spec)
项目规范的详细说明文档，包含：
- 功能需求
- 用户故事
- 验收标准
- 技术约束

#### 用户故事 (User Story)
从用户视角描述的功能需求。

#### 特性 (Feature)
功能的具体描述。

## 📖 工作流程

### 1. 需求收集阶段

- **目的**: 明确项目要做什么
- **活动**:
  - 与利益相关方沟通
  - 收集用户反馈
  - 分析现有代码
- **产出**: 需求文档、用户故事

### 2. 规范制定阶段

- **目的**: 将需求转化为具体规范
- **活动**:
  - 定义数据结构
  - 设计 API 接口
  - 确定技术方案
  - **产出**: 项目规范文档、类型定义

### 3. 开发实施阶段

- **目的**: 按照规范实施开发
- **活动**:
  - 编写代码
  - 编写测试
  - 代码审查
  - 文档编写
  - **产出**: 可工作的软件、文档

### 4. 测试验证阶段

- **目的**: 确保软件质量
- **活动**:
  - 单元测试
  - 集成测试
  - 性能测试
  - 用户验收测试
  - **产出**: 测试报告、质量报告

## 🎯 与 CatUI 的关系

### 配合方式

**Speckit** 和 **Specify** 工具相辅相成：

1. **Specify** (.specify/) - 专注于文档管理
   - 管理文档结构
   - 生成文档导航
   - 维护文档质量

2. **Speckit** - 专注于项目规范
   - 管理需求和用户故事
   - 跟踪开发进度
   - 生成规范文档

### 实际使用

对于 **CatUI** 项目，主要使用 **Specify** 工具进行文档管理：

```bash
# 示例：创建新的使用指南
specify create --type guide --id "ctable-usage" --title "CTable 组件使用指南"
```

## 📝 文档类型

### 指南文档 (Guide)

提供组件或功能的使用说明和示例。

### API 文档

详细的 API 接口说明，包括方法签名、参数、返回值等。

### 组件文档 (Component Doc)

特定组件的技术文档和实现细节。

## 💡 常见问题

### Q: Speckit 和 Specify 有什么区别？

**A**:
- **Specify**: .specify/ 目录，用于文档管理
- **Speckit**: .speckit/ 目录，用于项目规范管理

### Q: 我应该使用哪个工具？

**A**: 对于 **CatUI** 项目，主要使用 **Specify** 工具：
- 创建新文档：`specify create --type guide --id "xxx" --title "标题"`
- 清理旧文档：`specify cleanup --pattern "doc/old/**"`

### Q: 文档应该写多详细？

**A**:
- **概览**: 简要说明功能的用途和基本使用方式
- **详细内容**: 提供完整的 API 文档、代码示例、配置说明
- **示例**: 每个功能至少包含一个完整示例
